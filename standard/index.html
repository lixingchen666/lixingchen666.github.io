# H5项目规范

# 为什么要做规范

  1. 提高工作效率，便于相互协调工作。

  2. 实现代码一体化，无阻碍维护及修改。

  3. 项目结构清晰，代码有序，提升框架质量。

  4. 减少我们维护的时间成本

# PC端的项目目录结构

├── build                      # 构建相关
├── mock                       # 项目mock 模拟数据
├── plop-templates             # 基本模板
├── public                     # 静态资源
│   │── favicon.ico            # favicon图标
│   └── index.html             # html模板
├── src                        # 源代码
│   ├── api                    # 所有请求
│   ├── assets                 # 主题 字体等静态资源
│   ├── components             # 全局公用组件
│   ├── directive              # 全局指令
│   ├── filters                # 全局 filter
│   ├── icons                  # 项目所有 svg icons
│   ├── lang                   # 国际化 language
│   ├── layout                 # 全局 layout
│   ├── router                 # 路由
│   ├── store                  # 全局 store管理
│   ├── styles                 # 全局样式
│   ├── utils                  # 全局公用方法
│   ├── vendor                 # 公用vendor
│   ├── views                  # views 所有页面
│   ├── App.vue                # 入口页面
│   ├── main.js                # 入口文件 加载组件 初始化等
│   └── permission.js          # 权限管理
├── tests                      # 测试
├── .env.xxx                   # 环境变量配置
├── .eslintrc.js               # eslint 配置项
├── .babelrc                   # babel-loader 配置
├── .travis.yml                # 自动化CI配置
├── vue.config.js              # vue-cli 配置
├── postcss.config.js          # postcss 配置
└── package.json               # package.json

# 命名规范

  1. 文件文件夹命名遵循以下原则

    短横拼接法（kebab-case）

    ├── my-component                   
│   ├── index.vue

  2. 组件命名原则

    全局component：大驼峰命名

    全局组件放在src/components

    ├── src                  
│   ├── components
│   		├── MyComponent
│   				├── index.vue

    局部component：大驼峰命名

    局部组件组件放在当前页面文件/components

    ├── views                  
│   ├── my-page
│   		├── components
│   				├── MyComponent
│   						├── index.vue

  3. Js文件命名规范

    .js文件命名：短横拼接法

    my-script.js

    Api文件命名：统一添加Api后缀

    my-pageApi.js

  4. Vue文件命名规范

    .vue文件命名：小驼峰命名法

    myVuePage.vue

  5. Css文件命名规范

    .css/.less/.scss：短横拼接法

    xxx-xxx-xxx..css

  6. 图片命名规范

    图片命名：模块名称+具体页面名称+图片含义

    order-details-banner.png

  7. 代码变量和函数命名规范

    代码变量命名规范：常量必须大写

    函数命名命名规范：小驼峰命名法

    // 常量
const CONTENT = 'content'
// 函数命名
function addNumber(){}

  8. 路由传值规范

    路由传值： 禁止传整个对象，具体到对象属性

    this.$router.push({
	path:"/home",
	query:{name:data.name}
})

  9. 代码注释规范

    代码页面注释：第一行描述清楚页面是做什么用的

    <!-- 文档详情页 -->

  10. 代码注释规范 data中的规范：data中的属性明确声明的意义

  11. 代码细节

    • 所有的if都需要写括号，即使是一行代码
    • 所有的v-for都要带上key属性，增加遍历效率
    • 使用简化写法，比如v-bind:id,简化为:id
    • 判断语句、赋值语句适当的地方要加空格
    • Class使用小写字母横线分割
